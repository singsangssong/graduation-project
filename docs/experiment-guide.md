# 실험 가이드

## 실험 목적

실험 자동화의 목적은 제한된 환경에서 미들웨어 정책별 동작 차이를 반복해서 관찰하는
것입니다. 다음 질문을 확인할 수 있도록 구성되어 있습니다.

- 동일 자원 읽기를 융합했을 때 논리적 조회 횟수는 어떻게 달라지는가?
- 비용 점수를 사용하는 중재와 사용하지 않는 중재는 어떤 후보를 선택하는가?
- 동시 요청 수가 증가할 때 처리량과 지연시간은 어떻게 변화하는가?
- Saga 보상, 멱등성, 실패 기록과 재시작 복구가 동작하는가?

실험 결과는 단일 노드와 통제된 ticket-stock 워크로드에서 측정됩니다. 일반 서비스의
성능이나 실제 데이터베이스 I/O 절감 효과를 직접 나타내지 않습니다.

## 비교 모드

| 모드 | 읽기 정책 | 커밋 정책 |
| --- | --- | --- |
| `baseline` | 요청별 독립 논리적 조회 | 비용 비인지 순서 |
| `qcfuse` | 동일 키 읽기 융합 | 비용 비인지 순서 |
| `full` | 동일 키 읽기 융합 | 비용 점수 기반 순서 |

이 구성을 통해 읽기 융합과 비용 인지 중재를 분리해서 비교할 수 있습니다.

## 실행 프로필

| 프로필 | 용도 | 구성 |
| --- | --- | --- |
| `live` | 짧은 실행과 기능 확인 | 작은 동시성 규모, 1회 반복 |
| `paper` | 반복 실행과 통계 확인 | 여러 동시성 규모, 5회 반복 |

### Live 프로필

```sh
./scripts/run_live_experiment.sh
```

### 반복 프로필

```sh
./scripts/run_paper_experiment.sh
```

고정된 `latest-live`, `latest-paper` 디렉터리는 이전 실행의 SQLite 상태를 보존할 수
있습니다. 독립적인 실행이 필요하면 타임스탬프가 포함된 새 경로를 사용합니다.

```sh
python3 agent-python/experiment_runner.py \
  --profile live \
  --output-dir "outputs/experiments/live-$(date +%Y%m%d-%H%M%S)"
```

## 실행 과정

[`agent-python/experiment_runner.py`](../agent-python/experiment_runner.py)는 각 모드별로
다음 작업을 수행합니다.

1. 사용 가능한 로컬 포트를 선택합니다.
2. 모드별 환경 변수를 설정해 Go 미들웨어를 실행합니다.
3. 동일 seed로 에이전트 입력을 생성합니다.
4. 에이전트들이 읽기 요청 후 같은 commit barrier에서 경쟁하도록 합니다.
5. metrics를 수집하고 실행 결과를 저장합니다.
6. 별도의 Saga 신뢰성 및 재시작 복구 검증을 수행합니다.

## 측정 지표

| 지표 | 의미 |
| --- | --- |
| `throughput_tps` | 완료 요청 수를 실행 시간으로 나눈 값 |
| `p95_latency_ms` | 요청 지연시간의 95 백분위수 |
| `logical_db_reads` | 미들웨어 정책상 수행한 논리적 조회 수 |
| `io_reduction_pct` | 개별 조회 기준 논리적 조회 감소 비율 |
| `winner_protection_pct` | 선택된 후보 비용과 최대 후보 비용의 비율 |
| `error_rate_pct` | 오류가 발생한 요청 비율 |

`winner_protection_pct`는 정의된 비용 점수 정책과 선택 결과가 얼마나 일치하는지
관찰하는 값입니다. 작업의 실제 가치나 최적성을 나타내지 않습니다.

## 결과 파일

실험 출력 디렉터리에는 다음 파일이 생성됩니다.

| 파일 | 내용 |
| --- | --- |
| `raw_runs.csv` | 개별 반복 실행 결과 |
| `summary.csv` | 모드와 동시성 규모별 요약 |
| `summary.json` | 요약 데이터의 JSON 표현 |
| `reliability.json` | Saga 신뢰성 검증 결과 |
| `report.html` | 브라우저에서 확인할 수 있는 요약 보고서 |
| `server-*.log` | 모드별 Go 미들웨어 로그 |
| `*.db` | 실험에서 사용한 SQLite 상태 |

## 해석 시 주의사항

- 논리적 조회 카운터는 물리 DB I/O가 아닙니다.
- 동일한 `resource_id + intent` 요청이 집중되는 워크로드입니다.
- 실제 LLM API 호출 대신 재현 가능한 토큰·지연 입력을 사용합니다.
- 단일 머신의 스케줄링과 부하 상태가 처리량에 영향을 줄 수 있습니다.
- SQLite 단일 연결 구조이며 분산 DB 경합을 재현하지 않습니다.
- 비용 점수는 고정 가중치 기반 대리변수입니다.
- 반복 횟수와 워크로드를 확장한 별도 검증 없이 결과를 일반화하지 않습니다.
